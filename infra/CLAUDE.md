# Infra — Claude Code Context

Docker, Compose, Caddy, Mosquitto config, and optional LXC profiles for
deploying the Painting QC stack.

```

├── docker-compose.prod.yml      # RPi production deployment
├── .env.example                 # Documented env vars
├── caddy/
```

## Containerization principles

- **One image per component.** `qc-server`, `qc-dashboard`. Mosquitto and
  Caddy use upstream images unchanged.
- **Multi-arch builds.** Every image must build for `linux/amd64` (dev) and
  `linux/arm64` (RPi 4). Use `docker buildx`. Verify in CI.
- **Multi-stage Dockerfiles.** Builder stage compiles, runtime stage is slim.
  Final image runs as a non-root user.
- **Same image, different env.** Dev and prod compose files reference the
  same images; only env files and volume mounts differ. No "dev image" and
  "prod image" divergence.
- **Healthchecks mandatory.** Every service has a `HEALTHCHECK` in its image
  or `healthcheck:` in compose. `depends_on: condition: service_healthy`.
- **Restart policies.** `restart: unless-stopped` for all services in prod.
- **Logs to stdout/stderr.** No file logging from inside containers; Docker
  collects them.
- **Volumes for state, never bind mounts in prod.** Named volumes:
  `qc-data` (SQLite), `mosquitto-data`, `caddy-data`.
- **Networks are explicit.** A single `qc-net` bridge network. Services
  refer to each other by service name (`qc-server`, `mosquitto`), never IP.

## Compose pattern

`docker-compose.dev.yml`:
- Bind-mounts `../server` and `../dashboard` source for hot reload
- Exposes ports to host (8000, 5173, 1883)
- Uses `target: dev` stage in Dockerfile for development tooling
- `.env` file at `infra/.env`

`docker-compose.prod.yml`:
- No source bind mounts; pulls/builds images and runs them as-is
- Only Caddy exposes ports to host (80, 443)
- Server/dashboard reachable only via Caddy reverse proxy
- Mosquitto exposes 1883 to host network (STM32 devices connect)
- `.env` file at `/etc/qc/.env` on the RPi (root-readable only)

## Mosquitto

Single broker for the stack. Config in `infra/mosquitto/`:
- `mosquitto.conf`: persistence on, listener 1883, auth required
- `acl.conf`: per-username topic restrictions
  - `qc-server` username: full access to `qc/#`
  - Device usernames: each restricted to its own `qc/device/{id}/#` and
    read-only on `qc/config/#`
- Passwords generated by `scripts/provision-device.sh`

## Caddy

Reverse proxy and SPA host. Routes:
- `/` → dashboard container static files
- `/api/*` → server container
- `/api/ws/*` → server container with websocket upgrade
- `/config.js` → dynamic config injection from env

Dev: HTTP only. Prod: self-signed or internal CA cert; document trust setup
in `docs/deployment.md`.

## LXC alternative

Some plant IT departments prefer LXC over Docker for resource efficiency on
constrained hardware. We support running the entire Docker stack inside a
single LXC container as a middle ground.

Profile in `lxc/qc-stack.profile` provisions:
- An unprivileged container
- Docker + docker-compose installed inside
- The compose stack started on container boot
- Mosquitto port forwarded to the host

Trade-off: easier filesystem snapshots, slightly lower overhead, more
moving parts. Default for new deployments remains plain Docker on the host.

## Provisioning a new device

`scripts/provision-device.sh <device-id>` does:
1. Generate a random password
2. Add `qc-device-<id>:<password>` to Mosquitto's `passwd` file
3. Append ACL entries for that user
4. Reload Mosquitto (no restart, no lost connections)
5. Print credentials to be flashed into the STM32's QSPI

This is the only place device credentials are generated. Never hardcode them
in firmware or compose files.

## Build & verification

```bash
# Local dev (from repo root)
docker compose -f infra/docker-compose.dev.yml up --build

# Build production images for ARM64 (from CI or dev laptop)
docker buildx build --platform linux/arm64 -t qc-server:0.1.0 server/
docker buildx build --platform linux/arm64 -t qc-dashboard:0.1.0 dashboard/

# Deploy to RPi
./scripts/deploy.sh production
```

## Environment variables

See `.env.example`. Categories:
- **Secrets:** `JWT_SECRET`, `MQTT_PASSWORD`, etc. Never commit. Generate
  fresh per environment.
- **Hosts/ports:** all service-to-service comms use service names; only
  externally exposed ports matter
- **Feature flags (build-time):** `BUILD_FEATURE_*` — usually none needed
- **Plant config:** `PLANT_NAME`, `LOCALE`, `TIMEZONE`

## DO NOT

- Do not embed credentials in Dockerfiles or compose files
- Do not use `latest` tag in `docker-compose.prod.yml`; pin versions
- Do not run containers as root
- Do not expose Mosquitto, server, or dashboard ports directly in prod;
  go through Caddy except where the STM32 must connect to Mosquitto
- Do not use host networking unless absolutely required and documented
- Do not install packages into images at runtime; bake into the image
